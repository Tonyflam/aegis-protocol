## 7. What problem are you solving? (≤2000 chars)

**Problem:** BNB Chain processes more retail DeFi volume than any other EVM chain, but rug pulls, honeypots, contract upgrades, sell-tax manipulation, and protocol exploits drained an estimated **$2B+ across BSC in 2024–2025**. Existing solutions fail in three specific ways:

1. **Centralized scanners** are one-shot, reactive, and rely on heuristics that bad actors trivially bypass.
2. **Wallet alerts** fire *after* damage occurs and assume the user is awake, online, and decisive.
3. **Audit firms** are reactive and priced for protocols, not retail users.

Every existing tool ends at notification. Capital still depends on human reaction speed.

**Solution: Aegis Protocol — AI-native security infrastructure on BNB Chain.** Three integrated pillars sharing one autonomous agent and one on-chain trust anchor:

- **Scanner (free):** paste any BSC token → 0–100 risk score in 5 seconds. Bytecode analysis, LP-lock verification, PancakeSwap honeypot simulation, holder concentration, sell-tax measurement. Results cached on-chain to amortize cost.
- **Guardian (freemium):** connect wallet → agent monitors holdings every 30 seconds, even with the site closed. Detects whale dumps, liquidity pulls, sell-tax raises, proxy upgrades, owner privilege exercise. Critical alerts pushed via Telegram.
- **Vault (paid):** deposit BNB → 80% supplied to Venus Protocol for real lending APY, 20% instant-withdrawal buffer. Agent harvests yield, can emergency-withdraw on protocol risk signals, and triggers per-user stop-loss via PancakeSwap V2 when price thresholds breach.

**Why this is defensible:** every AI decision is hashed (keccak256) and written to `DecisionLogger` on-chain. Every premium feature is gated by `AegisTokenGate` reading $UNIQ balance. The agent operates only via permissioned contract methods — it physically *cannot* withdraw user funds to an external address. Non-custodial by construction, transparent by audit trail.

We're the first AI agent for BNB DeFi that doesn't just alert — it acts.

4. **Ecosystem alignment.** BNB Chain explicitly favors open builders. DappBay, MVB, Binance Alpha — every layer of ecosystem support assumes verifiable code.

5. **Composability.** Other BSC dApps can build on top of `AegisScanner` and `DecisionLogger` without permission. This is how a security primitive becomes infrastructure.

**What we keep proprietary:**
- AI prompt engineering parameters and active threat-classification tuning weights (kept in private env config) — these change weekly as new attack patterns emerge and don't need to be public.
- Operational keys, API keys, private node endpoints.

In short: code public, infrastructure private, community owned.

## 29. Which regulatory frameworks apply to your product, and what is your current approach to compliance? (≤2000 chars)

**Relevant frameworks (jurisdiction × product layer):**

**1. Token classification ($UNIQ — primary regulatory surface)**
- $UNIQ is structured as a **utility token**: fixed supply, no governance claim, no profit expectation, no team allocation, fair-launched, ownership renounced, LP locked.
- Avoids US Howey-test "common enterprise / expectation of profits" prongs by design.
- EU MiCA: $UNIQ falls under "utility token" definition (Art. 3(1)(9)) — disclosure obligations addressed in our public whitepaper and on-chain documentation.
- Approach: **proactive disclosure** — published whitepaper, on-chain renouncement proof, no marketing claims of price appreciation, "DYOR / not financial advice" disclaimers on all communications.

**2. Vault / yield product**
- US: structured to avoid "investment contract" classification — non-custodial (users always control withdrawal), yield is pass-through from Venus Protocol (no Aegis-controlled return promise), fees are explicit service charges not profit-sharing.
- EU MiCA: vault is closer to a "DeFi front-end" service; we follow MiCA Title V (CASP) developments and will adapt if scope expands.
- We do **not** geo-block currently but will implement geographic restrictions if required (US persons most likely affected).

**3. Data / privacy**
- No KYC at the scanner / guardian / vault layer (non-custodial, wallet-only)
- Telegram alerts: only chat ID stored, no PII
- GDPR-compatible by minimization

**4. AML/CFT**
- All vault flows are on-chain and traceable
- We monitor sanctioned-address lists (OFAC) and can block at the agent layer if required
- Open to integrating Chainalysis / TRM Labs screening at the agent level

**Long-term legal viability strategy:**
- Incorporate during residency in a clear Web3 jurisdiction (Cayman / BVI / UAE)
- Engage Web3-native legal counsel (intro via YZi Labs network is a key ask)

## 34. ONE moment of failure that taught a valuable lesson (≤2000 chars)

**April 12, 2026 — I had to publicly admit the original product wasn't useful.**

For two months after the hackathon, I had been bolting features onto the original Aegis design — agent NFTs, reputation systems, more registry tiers, fancier on-chain logic. I was shipping fast but I was shipping the wrong things. In my Telegram group, the same questions kept coming up: "When is staking?" "What does this actually do for me?" "I scanned my token, what now?"

I had built features that *I* thought were elegant. The community was politely telling me they were useless.

The failure moment was reading back my own roadmap and realizing I had been optimizing for technical sophistication, not user value. Agent NFTs were cool. Nobody was minting them. Reputation scores were elegant. Nobody was checking them. The actual painful problem — "I'm about to buy this token, is it safe?" "I hold these 12 tokens, are any of them about to rug?" — was buried under abstractions.

That night I drafted a post for the Telegram group titled (in Chinese and English): *"I've been quiet because I was rebuilding from scratch. Most of what I built isn't useful. Here's what's actually shipping next."* I listed the three concrete pillars: Scanner, Guardian, Vault. No agents-of-agents. No NFT identity. Just the product the community had been asking for the whole time.

I was terrified to post it. I was sure the holders would dump and leave.

The opposite happened. Engagement went up. Holders re-organized. The same names that had asked "are you abandoned?" started sharing addresses for me to scan as the new Scanner came online. The reset was the moment the project actually started.

**Lesson:** the user is the spec. Not the architecture diagram, not the whitepaper, not the founder's aesthetic preference. Anything you build that the user can't immediately use against their actual pain is dead weight, no matter how technically beautiful. Ship the painful obvious thing first. Make the elegant abstractions later.

This lesson now shapes every roadmap decision: each feature must answer the question "which sentence in our Telegram chat does this make obsolete?" If it doesn't, it doesn't ship.
